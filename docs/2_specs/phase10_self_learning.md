## Purpose

Today the AI extracts facts and the user reads them, but has no way to say "this is right," "this is wrong," or "here is what I actually know." Phase 10 closes that feedback loop. After this phase, the system can:

1. **Record corrections** — confirm, reject, or revise individual facts, with the reason tracked and the user's trust reflected in a per-fact score.
2. **Protect validated facts** — once a user confirms a fact, the classify pipeline no longer silently overwrites it. Instead it writes a new competing entry, surfacing the conflict.
3. **Learn from mistakes** — past AI extraction errors are fed back into the extraction prompt as teaching examples, improving future accuracy.
4. **Synthesize reports** — on-demand health summaries (correction trends, coverage gaps, conflicts) generated by AI and stored for history.
5. **Accept annotations** — lightweight comments on knowledge entries that the AI considers during future extraction runs.

No frontend. The consumer AI (Claude Desktop) drives everything through MCP tools.

## Feature overview

**The feedback loop, in plain English:**

A user tells the consumer AI "this fact is wrong." The consumer AI checks the source document, determines whether the AI misread it (`ai_error`) or the source was outdated (`stale_source`), then calls the Correction Tool with the verdict. The system records the correction, adjusts the fact's trust score (down for rejections, up for confirmations), and logs everything to the audit trail.

Next time the classify pipeline runs, two things are different:
1. **The overwrite guard** checks trust before updating any existing fact. If the user already confirmed a fact (trust > 0.5), the pipeline cannot silently overwrite it — it writes a new competing entry instead, creating a visible conflict.
2. **The few-shot injector** pulls recent `ai_error` corrections for the dimension being processed and injects them as "don't make this mistake again" examples into the extraction prompt.

Separately, the user can attach comments to any fact (lightweight annotations the AI sees during future extraction), and can request on-demand reports that synthesize knowledge health, correction trends, coverage gaps, or conflicts.

**Edge cases:**
- A `revise` operation retires the old entry and creates a fresh entry at trust 0.6 with the corrected text. The old entry's correction history stays attached to the retired row.
- When the overwrite guard blocks a write, both the trusted original and the new pending entry coexist — the "conflicts" report type detects this pattern.
- Retired entries remain in the search index (bug fix) so "what did we used to think about X?" still finds them. They are excluded from context injection at query time by the existing `WHERE status != 'retired'` filter.

- **Multiple learning signal types** — `fact_corrections` covers fact corrections only. Future signal types (classify errors, summary corrections) get their own tables.

## Open Question Resolutions

| OQ | Question | Resolution | Rationale |
|---|---|---|---|
| OQ-P10-01 | Should trust_score reset to 0.5 on entry update? | **Preserve.** `write_entries()` update path reads the existing entry and merges sources. Trust_score must be copied from the existing row, not reset. The `KnowledgeEntry` constructed for `ke_upsert()` must carry the existing trust_score. | Trust represents user validation, not AI confidence. An AI update to fact text should not erase user trust. |
| OQ-P10-02 | Should corrections on a revised entry be inherited by the new entry? | **No inheritance.** New entry starts clean. Old entry's corrections remain attached to the retired row. | Keeps the model simple. The correction log is a history of what happened to each specific entry. | OQ-P10-03 | Should `kms_reports` cache or always regenerate? | **Always regenerate.** Each call creates a new row in `reports` table. | Simplest correct behavior. LLM cost is acceptable for single-user. Add `max_age_hours` config per report type later if needed. |
| OQ-P10-04 | Should few-shot corrections be re-injected on self-correction retry? | **Yes, same as first attempt.** Few-shot examples are static per dimension and do not change between retries. The `previous_attempt_feedback` variable handles retry-specific context. | Few-shot examples teach general extraction accuracy. Retry feedback teaches specific-to-this-document corrections. | OQ-P10-05 | Should comments have a character cap? | **No cap.** Context loader's per-dimension budget naturally truncates. | Premature optimization. If abuse is observed, add a column-level cap later. |

### 4. Trust Calculator

**Goal.** A pure function that computes the new trust score after a correction operation. No side effects, no DB access, no hardcoded floats.

**Build.** Create `src/pipelines/trust.py` with:

`adjust_trust(current_score: float, operation: str, config: SelfLearningConfig) -> float`

- `confirm`: `min(1.0, current + config.trust_confirm_delta)`
- `reject`: `max(0.0, current + config.trust_reject_delta)`
- `revise`: returns `config.trust_revise_base` (applied to new entry)
- Unknown operation: returns `current` unchanged (defensive)

Returns `Result[float]` per C-12.

**Depends on.** Component 3 (config extension — reads deltas from `SelfLearningConfig`).

**Interface shape.** Callers pass the current trust score and operation string; receive the new score. No DB, no side effects. Pure input/output.

**Done when.** `adjust_trust(0.5, "confirm", config)` returns `0.55`. `adjust_trust(0.5, "reject", config)` returns `0.40`. `adjust_trust(0.5, "revise", config)` returns `0.6`. `adjust_trust(1.0, "confirm", config)` returns `1.0` (capped). `adjust_trust(0.0, "reject", config)` returns `0.0` (floored). No float literals in the function body.

**Behaviors covered.** P10-SL-03 (trust deltas), P10-SL-05 (adjust_trust pure function, config-driven).

### 5. Overwrite Guard activation

**Goal.** When a user has confirmed a fact (trust > threshold), the classify pipeline cannot silently overwrite it. Instead it writes a new pending entry, creating a visible conflict.

**Build.** Modify `_should_overwrite()` in `classify_writer.py`:
- Change signature to accept `existing_entry` and `*, threshold: float` keyword argument.
- Return `existing_entry.trust_score <= threshold`.
- At both call sites (update path ~line 212, twin-fold path ~line 277), read threshold from `CONFIG.main.self_learning.overwrite_trust_threshold` and pass it.
- When `_should_overwrite()` returns `False`: skip the update/fold, instead insert the new fact as a fresh `pending` entry with the same dimension/entity/tag. Log `overwrite_blocked entry_id=X trust=Y threshold=Z`.

**Depends on.** Component 3 (config extension — reads threshold), Component 4 (Trust Calculator — the trust values that trigger this guard come from corrections).

**Assumes.** A1 (_should_overwrite seam exists with expected signature and call sites), A9 (trust_score is currently inert at 0.5, so activating the guard with threshold 0.5 means strict-greater blocks only user-confirmed entries).

**Decisions.**
- Q: Should the threshold comparison be `>` or `>=`? Leaning `>` (strict greater) because trust 0.5 is the unconfirmed default — entries at exactly 0.5 should remain freely updatable. Only entries confirmed by a user (trust > 0.5) should be protected.

**Done when.** An entry with trust_score 0.6 is not overwritten by classify — a new pending entry is created alongside it. An entry with trust_score 0.5 IS overwritten normally. The `_should_overwrite` function body contains no float literals (reads threshold from parameter).

**Behaviors covered.** P10-SL-06 (overwrite guard activation), P10-SL-07 (coexisting trusted + pending entries).

### 7. Few-shot Correction Selector

**Goal.** Select the most relevant past AI errors to teach the extraction prompt for the current dimension.

**Build.** Create `src/pipelines/few_shot.py` with two public functions:

`select_corrections(dimension: str, doc_entities: list[str], *, cap: int, db_path: Path | None = None) -> Result[list[dict]]`

Selection algorithm:
1. Query `fact_corrections` WHERE `reason_category = 'ai_error'`, joined with `knowledge_entries` for dimension/entity.
2. Score each correction: +3 dimension match, +2 entity overlap with `doc_entities`, +1 base recency (position penalty from `ORDER BY created_at DESC`).
3. Sort by score descending, take top `cap`.
4. Return list of dicts: `{old_fact, new_fact, feedback, dimension, entity}`.

`format_few_shot(corrections: list[dict]) -> str`

Formats corrections as teaching text:
```
Previous extraction mistakes to avoid:
- For [entity] in [dimension]: The AI incorrectly extracted "[old_fact]". The correct fact is "[new_fact]". [feedback if present]
```

Returns empty string if corrections list is empty (no examples to inject).

**Depends on.** Component 1 (migration — `fact_corrections` table must exist).

**Interface shape.** `select_corrections` reads DB, returns structured dicts. `format_few_shot` is pure string formatting. Classify orchestrator calls both, passes result to `extract()`.

**Done when.** Given 10 ai_error corrections in the DB, `select_corrections("people", ["Anthony"], cap=5)` returns 5 corrections with people-dimension and Anthony-entity corrections ranked highest. `format_few_shot` produces non-empty text for non-empty input, empty string for empty input. `stale_source` corrections are never returned.

**Behaviors covered.** P10-SL-10 (relevance-ranked selection), P10-SL-11 (stale_source excluded), P10-SL-12 (config-driven cap).

### 10. Correction operation aliases and partial wiring in Correction Handler

**Goal.** The Correction Handler understands the Phase 10 terminology (confirm/reject/revise) alongside the P9 names, and performs basic trust adjustment. Full correction recording (with reason_category, feedback, and fact_corrections table insert) is deferred to Slice 2 to avoid coupling with P9's still-evolving correct_entry internals.

**Build.** In `_correct.py:correct_entry()`:
- Add operation aliases: `confirm` maps to the same logic as `promote`, `reject` maps to `retire`, `revise` maps to `edit_fact` + retire old + create new at trust 0.6.
- After applying the operation, call `adjust_trust()` to compute new trust score and write it to the entry via `upsert()`.
- The `revise` flow: retire old entry, create new `KnowledgeEntry` with new fact text and trust = `adjust_trust(old_trust, "revise", config)`, upsert the new entry.

**Depends on.** Component 4 (Trust Calculator).

**Assumes.** A2 (correct_entry signature and operations).

**Decisions.**
- Q: Should `revise` preserve the old entry's sources list on the new entry? Leaning yes — the revised fact is still about the same sources, just corrected text.

### 15. Report synthesis pipeline

**Goal.** On-demand knowledge health reports generated by AI, stored for history, returned to the consumer.

**Build.**

1. **Report definitions config:** Create `config/reports.yaml` with 5 default report types:
   - `correction_summary` — correction counts, most-corrected entities, accuracy trend
   - `knowledge_health` — total facts by dimension, pending count, coverage metrics
   - `volatile_entries` — entries with correction count above volatility threshold
   - `coverage_gaps` — documents with zero knowledge entries extracted
   - `conflicts` — entries where overwrite guard blocked (same entity+dimension+tag with both trusted and pending status)

   Each report type has: `title`, `filters` (dimensions, entities, status, tags), `prompt` (synthesis instruction), `sources` (which corpora to read: facts, summaries, corrections).

2. **Pipeline module:** Create `src/pipelines/reports.py` with:
   `synthesize_report(report_type: str, *, config: MainConfig, db_path: Path | None = None) -> Result[dict]`

   Steps:
   1. Load report definition from `config/reports.yaml`.
   2. Gather data per `sources` config: query `knowledge_entries` and/or `documents` and/or `fact_corrections` using the filter constraints.
   3. Format gathered data as context text.
   4. Render the report's prompt template with the context.
   5. Call `get_provider("synthesis", config)` for the LLM synthesis.
   6. Store the result in the `reports` table (report_type, title, body, prompt_used, filters_used as JSON, sources_used as JSON array of IDs).
   7. Return `Success({"report_id": id, "title": ..., "body": ...})`.

3. **Conflict detection (for "conflicts" report):** Query for same entity + dimension + tag combinations where at least one entry is `confident` (trust > default) and another is `pending`. This is the structural heuristic from D3.

**Depends on.** Component 1 (migration — `reports` table), P9 context assembly patterns (reuse data gathering logic).

**Interface shape.** Callers pass a report type string; receive a synthesized report dict. The report definition is data (YAML), not code. Adding a new report type requires zero code changes (extension point rule).

**Dependency category.** local-substitutable — test with in-memory DB and mock LLM provider.

**Done when.** `synthesize_report("knowledge_health", config=config)` returns a Success with a report body synthesized by the LLM. The report is stored in the `reports` table. Adding a new entry to `reports.yaml` with different filters produces a different report without code changes. The "conflicts" report correctly identifies entries blocked by the overwrite guard.

**Behaviors covered.** P10-SL-13 (on-demand synthesis), P10-SL-14 (extension point — YAML only), P10-SL-15 (5 default report types), P10-SL-16 (conflicts report detects overwrite-guard blocks).
